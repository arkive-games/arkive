## Problem

`creature{Intellect,Feral,Nature,Trans,Special}`) is fully wired in the frontend
and in `data/types.json`, but **no creature markers exist in `data/markers/*.json`**,

## Key discovery â€” the full chain is in the raw export

`Data/Table/*.json` + the per-map `MapData.json`, verified end-to-end by tracing

## Approach

Derive everything in **`tools/`** (the only layer that sees the raw export) and
emit clustered creature markers into `data/markers/*.json`. The frontend stays a
pure renderer â€” **no frontend changes, no `types.yaml` changes** (the category,
five subtypes, and their `images/Creature_*.webp` icons already exist; the icons
are present in `frontend/public/images/`).

Rejected alternatives:
- *Export the pets the old `pets.py` uploaded to the backend* â€” explicitly out;
  contradicts the Phase 2 static-data contract.
- *Re-scrape the `G:\...\å® ç‰©` screenshot set* â€” not raw data; the curated
  `table.csv`/`table.pets.json` aren't even in the repo anymore.
- *Exhaustive (one marker per spawn point, ~6,842/map)* â€” too dense; clustering
  was chosen.
- *Per-pet portraits as the marker icon* â€” deferred (see Out of scope).

## Changes

### 1. tools â€” new module `tools/aion2/tools/maps/creatures.py`

Isolates the join + clustering so `extract.py` stays focused, and is unit-testable
without the full pipeline. Two pure functions:

- `build_pet_source_index(vehicle_list, item_table, npc_loot, l10n) â†’ dict[int, PetSource]`
  where `PetSource = {subtype, name_en, name_zhCN}`:
  - subtype = `ECreatureType::<T>` â†’ `creature<T>` (e.g. `Intellect â†’ creatureIntellect`).
  - Join `VehicleList.SoulItemName â†’ Item.ID â†’ NpcLoot.VehicleSoulItemId â†’ NpcId`.
  - Skip pets with `SoulItemName == "None"` (the 38 shop/Special pets).
  - Name from `VehicleList.Desc.Key` via `l10n` (en + zh-CN).
  - Returns a map keyed by **source `NpcId`** â†’ its pet.

- `cluster_creature_markers(spawn_info_list, transform, index, radius=200) â†’ list[dict]`:
  - For each spawn entry, for each `NpcId` in `NpcIdList` that is in `index`, add
    all `Positions[]` (worldâ†’pixel via `transform`) to that pet's point list.
  - Per pet: **deterministic greedy** clustering â€” sort points by `(x, y)`, then
    merge each point within `radius` px of an existing cluster centroid into it
    (running centroid); else start a new cluster.
  - Emit one marker dict per cluster:
    `{kind: "creature<T>", px: [cx, cy], name_en, name_zhCN, count: <points in cluster>}`.

`radius` defaults to **200**; expose it as a module constant for easy tuning.

### 2. tools â€” `extract.py` wires it in

In `extract_map`, after the existing `SpawnInfoList` pass that produces
`WorldMarkers` (the boss branch, ~L649):

- Build the index once per run (cache it; reuse `_table("VehicleList.json")`,
  `_table("Item.json")`, `_table("NpcLoot.json")` and the existing `l10n`).
- Call `cluster_creature_markers(spawn_info_list, transform, index)` and
  **append** the returned dicts to the map's `WorldMarkers` list (they share the
  `kind`/`px`/`name_en`/`name_zhCN` shape, plus `count`).

No change to how `WorldMarkers` is serialized into `parsed_data/maps/<Map>.json`.

### 3. tools â€” `emit_frontend.py` routes the new kinds

`tools/aion2/tools/maps/emit_frontend.py`:

  ```py

  "creatureSpecial": "creature",
  ```

  - `desc_en = f"{w['count']} spawn points"`, `desc_zhCN = f"{w['count']} å¤„åˆ·æ–°ç‚¹"`
    (zh-TW via the existing OpenCC pass). So the merge isn't silently lossy.

### 4. frontend / types / icons â€” no changes

`GameMarker.tsx` already renders `category === "creature"` as the circular pin
using `marker.icon || sub.icon`; with no per-marker icon set, each marker shows

`MarkerTypes.tsx` shows the PawPrint section (hidden by default, toggleable,

## Scope

- **Maps:** World_L_A first (build + verify live), then World_D_A. Same code; only
  the extracted map list differs. Other maps are out of scope for this pass.
- **Regeneration:** run the extract step then the emit step to rewrite
  `data/markers/World_L_A.json` (+ its locale files). `tools/` and `data/` are
  independent git repos, committed separately.

## Edge cases (expected, not errors)

- 38 shop/Special pets with `SoulItemName:"None"` â†’ no markers.
- Pets whose source creature spawns only in dungeons/instance-layer maps â†’ no
  marker on the open-world map.
- A spawn entry's `NpcIdList` may include non-pet NPCs â†’ ignored (not in index).
- Duplicate/again-identical positions â†’ absorbed by clustering; `emit_frontend`
  already hard-fails on duplicate marker ids.

## Out of scope (YAGNI)

- **Per-pet portrait icons.** `Portrait/Portrait_Vehicle` textures exist in the raw
  UI and could later be emitted + set as `marker.icon` per pet; v1 uses the five
  type-icons.
- Maps beyond World_L_A / World_D_A; abyss creature spawns.
- A sidebar count badge or per-cluster spawn-point list beyond the description text.
- Region linkage for creature markers (omitted, like teleport/boss).

- **Unit â€” clusterer:** crafted point sets â†’ assert cluster counts/centroids at a

- **Integration:** run extract+emit for World_L_A â†’ assert creature markers exist
